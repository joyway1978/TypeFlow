## Approaches Considered
### Approach A: 纯规则校验 + 语音播放（Minimal MVP）
- 语音播放：后端返回音频 URL 或直接生成音频并由前端播放
- 校验：后端把用户文本标准化（大小写、标点可选忽略、空格归一），再做逐词对比。

Effort: S
Risk: Low
Pros:
- 最快落地，校验逻辑可控且可解释
- 延迟低，不依赖复杂 AI 评估
Cons:
- 对“近似正确但写法不同”的容忍度需要你定义规则
Reuses:
- 直接复用通用文本对齐/差异算法（如 Python difflib 或自定义 token compare）

### Approach B: 基于轻量对齐 + 可配置容错策略（Ideal-ish）
- 校验：tokenization（按词或子词）+ 编辑距离对齐（允许小错误）
- 反馈：输出“对用户输入最可能的正确对齐路径”，并标红/标黄差异。

Effort: M
Risk: Med
Pros:
- 反馈更“像老师批改”，容忍常见偏差
- 用户体验更自然，不会被标点/大小写等细节打击
Cons:
- 需要你定义 token 规则与容错边界
Reuses:
- Python 中的编辑距离算法或 difflib，并加可配置策略层

### Approach C: 评测交给 LLM（Highest Flexibility）
- 校验：把句子、用户输入、朗读进度交给后端 LLM
- 反馈：LLM 输出纠错建议和高亮差异。

Effort: L
Risk: High
Pros:
- 能处理同义表达、拼写变体、部分语法错误
- 对未知输入更“宽容”
Cons:
- 成本和一致性不可控，可能出现判错/幻觉式反馈
Reuses:
- Python 服务层封装 LLM 调用与提示模板

## Recommended Approach
**采用 Approach B**（基于 token 对齐 + 编辑距离容错）。

理由：
- 你明确要“整句输入校验”，这通常需要良好的对齐策略，否则反馈要么太严格（用户全错），要么太笼统（无法学习）。
- Approach A 虽然更快，但你很快会遇到“用户输入与标准句子不完全一致但整体正确”的情况；那时你会重写校验层。B 把容错策略提前设计好。

## Validation Rules (Locked)
- 统一忽略：标点（punctuation）与大小写（case）。
- tokenization：以“按空格切词”为主，标点在进入 tokenization 前先被移除/统一。
- 对齐：采用 token level 编辑距离对齐（允许少量替换/删除/插入），生成最优对齐路径。
- 反馈：
  - 完全匹配的 token 标为正确。
  - 只因大小写/标点导致差异的 token 标为“忽略差异，视为正确”。
  - 其它 token 差异以红色高亮，并在旁边显示“标准 token → 你的 token”。
- 边界：对齐冲突时，以标准句子的 token 顺序为基准输出最短编辑路径（避免多解）。

## Success Criteria
- 用户点击“开始/播放”后 1-5 秒内能听到句子
- 用户输入完整句子并提交后，页面能展示：
  - 每个词/字符的正确性
  - 明确指出错误位置
  - 错误高亮规则与“忽略标点+忽略大小写”的策略一致
- 同一条句子下，多次练习反馈稳定一致
